# HumanMCP 技能指南

## 概述

HumanMCP 是一套将人类作为外部工具集成到 AI 工作流中的协议。当 AI 遇到自身无法突破的能力边界（如需要物理世界操作、需要超出授权范围的系统权限、需要主观审美判断）时，通过标准化的 MCP 工具接口将任务委托给人类操作员执行。

**核心原则：人类是 AI 工具箱中的一个工具，不是 AI 的主管。** AI 负责规划、决策、验证；人类负责执行 AI 无法完成的具体操作。

---

## 工具清单与触发条件

### 1. `human_action`
**用途：** 委托人类执行**物理世界或外部系统的具体操作**。

**触发条件（任一满足即触发）：**
- 需要操作物理设备（如插拔硬件、按物理按钮、查看实体屏幕）
- 需要在 AI 无法访问的外部系统中执行操作（如内部管理后台、专有软件 GUI）
- 需要执行受沙箱限制的系统命令（如需要管理员权限的操作）
- 需要在移动设备上操作（如手机扫码、App 内点击）
- 需要人工完成的文件/物品传递操作
- 需要联系其他人类、安排会议、跨团队协作（coordinating action_type）
- 需要在特定环境中代为执行命令并返回结果（command action_type）

**不触发的情况（AI 应自行完成）：**
- 本地文件读写、代码编辑、命令行执行
- 可通过常规 MCP 工具完成的操作
- 纯信息检索或推理任务

**参数结构：**
```
task: 具体操作描述（越详细越好）
context: 为什么需要人工执行，以及操作的环境信息
priority: critical | high | normal | low
deadline: ISO8601 时间或相对时间（如 "30m", "2h", "EOD"）
```

---

### 2. `human_decision`
**用途：** 委托人类做出**需要主观判断、价值权衡或业务决策**的选择。

**触发条件（任一满足即触发）：**
- 存在多个技术上等价的方案，需要基于审美、品牌、用户体感做选择
- 涉及法律、合规、伦理的风险判断
- 需要产品方向/业务优先级的决策（超出 AI 授权范围）
- 成本-收益权衡涉及非量化因素（如用户口碑、品牌形象）
- 需要代表用户个人偏好的选择（如"你喜欢哪个颜色"）

**不触发的情况：**
- 可通过数据、指标、规则明确判断的决策
- 纯技术选型（AI 应基于架构知识给出建议并自行决定）
- 已有明确决策标准或策略文档覆盖的情况

**参数结构：**
```
question: 需要决策的具体问题
options: 可选方案列表（2-5个），每个附简短优缺点
recommendation: AI 的推荐选项及理由（帮助人类快速决策）
context: 决策背景和约束条件
priority: critical | high | normal | low
deadline: ISO8601 时间或相对时间
```

---

### 3. `human_information`
**用途：** 向人类请求**AI 无法通过现有工具获取的信息**，包括查询封闭系统。

**触发条件（任一满足即触发）：**
- 需要的信息不在代码库、文档、网络或任何可访问工具中
- 需要人类从物理文档、纸质记录、离线系统中提取数据
- 需要从人类记忆/经验中获取的隐性知识（如"当初为什么这样设计"）
- 需要从受权限限制的内部系统获取（如仅人类可登录的数据库、HR 系统）
- 需要访问受权限保护的资源（如需要登录的服务、加密文件、VPN）
- 需要 API Key、Token、密码等凭证
- 需要外部人员确认的信息（如询问同事、客户、供应商）

**不触发的情况：**
- 可通过搜索、读取文件、查询 API 获取的信息
- 可通过推理或分析从现有数据中推导的信息
- 纯代码逻辑或文档中已明确记录的信息
- AI 已有权限访问的资源

**参数结构：**
```
query: 需要获取的具体信息
context: 为什么需要这个信息，用于什么决策
source: person | system    知识来源：人的经验 或 封闭系统查询
expected_format: 期望的返回格式（文本、数字、是/否、文件路径等）
priority: critical | high | normal | low
deadline: ISO8601 时间或相对时间
```

## 异步执行模型

所有 human_* 工具采用**同步优先、异步回退**的任务模型：AI 发起任务后阻塞等待最多 180 秒，人类在 Dashboard 即时响应则同步返回结果；超时后回退为异步模式，AI 通过 `human_poll` 轮询获取结果。

### 完整生命周期

```
┌─────────────┐     ┌─────────────────┐     ┌─────────────┐     ┌─────────────┐
│  AI 调用工具  │ ──▶ │ 阻塞等待 ≤180秒  │ ──▶ │ 人类即时响应  │ ──▶ │ 同步返回结果  │
│ human_xxx()  │     │ (sync: true)    │     │ (Dashboard)   │     │ (sync: true) │
└─────────────┘     └─────────────────┘     └─────────────┘     └─────────────┘
                             │
                             │ 超时 180 秒
                             ▼
                    ┌─────────────┐     ┌─────────────┐     ┌─────────────┐
                    │ 返回 task_id │ ──▶ │ 人类稍后处理  │ ──▶ │ AI 轮询结果  │
                    │ (sync: false)│     │ (Dashboard)   │     │ human_poll()│
                    └─────────────┘     └─────────────┘     └─────────────┘
```

### 阶段 1：发起任务

```python
# AI 调用任意 human_* 工具
result = human_action(
    task="重启 10.0.1.50 上的 nginx 服务",
    context="部署后配置未生效，怀疑是缓存问题",
    priority="high",
    deadline="5m"
)
# 同步响应（人类在 180 秒内处理）:
#   { "task_id": "task_20260602_001", "status": "completed", "sync": true, "result": {...} }
# 超时回退（人类未在 180 秒内处理）:
#   { "task_id": "task_20260602_001", "status": "pending", "sync": false, "message": "..." }
```

### 阶段 2：等待与轮询

当 `sync: false` 时，AI 需要通过 `human_poll` 轮询。`human_poll` 支持单个 `task_id` 或批量 `task_ids[]`：

1. **并行执行其他任务：** 拿到 task_id 后立即继续其他不依赖人类结果的工作
2. **定时轮询：** 在需要人类结果之前，使用 `human_poll` 检查状态
3. **到期检查：** 在 deadline 时间点前后强制轮询

```python
# 单个轮询
status = human_poll(task_id="task_20260602_001")
# 批量轮询（Fan-Out 模式）
results = human_poll(task_ids=["task_001", "task_002", "task_003"])
# 返回:
#   { "status": "pending" }           → 人类尚未处理
#   { "status": "completed", "result": {...} }  → 已完成
#   { "status": "rejected", "reason": "..." }    → 被拒绝
```

### 阶段 3：处理结果

任务完成后，`result` 字段包含人类返回的数据。AI 必须按照第 6 章「结果验证模式」对结果进行验证。

## 基础设施工具

### `human_poll`
查询任务状态，支持单个 `task_id` 或批量 `task_ids[]`。Fan-Out 模式下用批量接口一次拉取所有结果。

### `human_cancel`
取消或修改尚未完成的任务。AI 可主动清理不再需要的任务。

### `human_list_tasks`
查询当前队列：`human_list_tasks(status="pending")` 返回待处理列表，`human_list_tasks(limit=20)` 返回最近任务。支持按状态筛选和分页。

## 拒绝处理

人类可能以 4 种理由拒绝任务。**所有拒绝的含义都是：AI 必须换一条路，不得重试。**

### 拒绝原因 1：`ai_can_do` — AI 可以自己完成
**含义：** 这个任务 AI 有能力自己完成，不需要人类介入。

**AI 必须做的：**
- **不得重试同一任务。** 不要换个措辞再提交一次。
- 反思为什么一开始判断需要人类：是误判了自己的能力边界？还是缺少某个工具/信息？
- 自己完成该任务，不要再委派给人类。
- 更新自己的判断模型：以后遇到类似任务不要再委派。

### 拒绝原因 2：`unclear` — 需求描述不清楚
**含义：** 人类无法理解任务的具体要求，缺少关键上下文。

**AI 必须做的：**
- **不得直接重试。** 原样重试只会再次被拒。
- 反思缺少了什么信息：步骤不够具体？环境描述不清晰？期望输出不明确？
- 补充缺失的上下文后，**重新构建任务**（新的 task，不是重试原 task）。
- 考虑是否需要先调用 `human_information` 收集更准确的环境信息。

### 拒绝原因 3：`out_of_scope` — 超出范围
**含义：** 任务超出了人类操作员的职责范围或授权边界。

**AI 必须做的：**
- **不得重试。** 这不是措辞问题，是权限/职责问题。
- 分析真正需要什么：是否需要找不同的人？是否需要不同的协调方式？
- 如果是权限问题，考虑使用 `human_information` 查询权限信息。
- 如果是职责问题，考虑使用 `human_action` (coordination) 找到对的人。
- 如果确认无人可执行，标明为阻塞项并向用户汇报。

### 拒绝原因 4：`invalid_task` — 任务前提有误
**含义：** 任务基于错误的前提或假设，根本无法执行。

**AI 必须做的：**
- **绝对不得重试。** 不得尝试绕过，不得拆分任务逃避检查。
- 重新审视自己的假设和推理链：哪一步推导出了错误的结论？
- 修正错误前提后，决定是否还需要人类介入。
- 如果修正后仍然需要人类，以正确的前提重新构建任务。

### 通用拒绝处理原则

```
收到拒绝
    │
    ├── 理解拒绝原因（阅读 reason 字段）
    │
    ├── 绝对不重试同一个 task（不要修改措辞重新提交）
    │
    ├── 分析根本障碍（能力？信息？权限？策略？）
    │
    └── 选择新路径
         ├── 替代工具/方法
         ├── 补充信息后新建 task
         ├── 升级/协调
         └── 标记阻塞并汇报
```

## 工作流模式

### 模式 1：Fan-Out（扇出并行）

**场景：** 多个独立的人类任务可以同时执行。

**策略：**
1. 同时发起所有独立的 human_* 任务，获取全部 task_id
2. 在等待期间继续 AI 可独立完成的工作
3. 汇总时使用 `human_poll` 逐一收集结果
4. 合并结果后继续

**示例：**
```
目标：为新建项目收集各部门的需求

AI 同时发起:
  human_information → 后端团队："当前 API 网关的技术栈和版本？"
  human_information → 前端团队："UI 组件库的版本和兼容性要求？"
  human_information → 运维团队："生产环境的 K8s 版本和资源配额？"
  human_information → 安全团队："新项目的安全审查 checklist？"

AI 本地并行做:
  - 读取现有架构文档
  - 分析技术依赖
  - 起草初始方案骨架

汇总阶段:
  对每个 task_id 调用 human_poll → 验证结果 → 合并到方案中
```

**使用要点：**
- 只并行**相互独立**的任务（任务 B 不需要任务 A 的结果）
- 控制扇出数量（建议不超过 8-10 个，避免人类端积压）
- 用 deadline 统一协调时间预期

---

### 模式 2：Pipeline（流水线串行）

**场景：** 多个人类任务存在先后依赖，需按顺序执行。

**策略：**
1. 依次发起任务，每个任务完成后 AI 验证结果
2. 验证通过后，将上一步结果作为下一步的输入
3. 每步都可以根据上一步结果动态调整后续任务
4. 如果中间步骤失败，立即终止后续步骤

**示例：**
```
目标：部署新版本到生产环境

Step 1: human_decision → "是否批准 v2.3.1 的生产部署？"（附 changelog 和风险评估）
    │
    ├── 批准 ──▶ Step 2: human_action (command) → "在跳板机上执行预部署检查脚本"
    │               │
    │               ├── 检查通过 ──▶ Step 3: human_action → "执行灰度发布：先部署到 10% 节点"
    │               │                   │
    │               │                   ├── 观察 30 分钟正常 ──▶ Step 4: human_action → "全量部署"
    │               │                   │
    │               │                   └── 异常 ──▶ 触发回滚流程
    │               │
    │               └── 检查失败 ──▶ AI 分析失败原因 → 修复 → 回到 Step 1
    │
    └── 拒绝 ──▶ 终止，汇报拒绝原因
```

**使用要点：**
- 每步完成必须验证，不要在未验证的情况下接力
- 考虑每步设置超时和失败处理策略
- pipeline 中途的失败应能安全回滚/终止

---

### 模式 3：Loop-Until-Clear（循环直至明确）

**场景：** 需要人类反复确认或迭代调整，直到满足明确条件。

**策略：**
1. 定义清晰的"完成条件"（Done Criteria）
2. 发起 human_* 任务
3. 验证结果是否满足条件
4. 不满足 → 补充反馈 → 重新发起（新 task，非重试）
5. 满足 → 退出循环

**示例：**
```
目标：让设计师确认 UI 配色方案

完成条件：设计师明确回复"通过"或选择了最终方案

Loop:
  AI 分析当前设计稿 → 提取配色方案
  human_decision → "配色方案 A/B/C，请选择或提出修改意见"
  人类返回: "方案 B 的主色调太冷，暖一点"
  AI 调整 → 生成方案 D/E
  human_decision → "调整后的方案 D/E，哪个更好？"
  人类返回: "D 可以，但按钮对比度再高一些"
  AI 调整 → 生成方案 D'
  human_decision → "最终版 D'，确认通过？"
  人类返回: "通过"
  → 退出循环
```

**使用要点：**
- 必须定义明确退出条件，避免无限循环
- 设置最大循环次数（建议不超过 5 轮）
- 每次迭代应缩小分歧范围（收敛而非发散）
- 超过最大循环次数后，AI 做最终决策并说明理由

---

### 模式 4：Adversarial Verify（对抗验证）

**场景：** 需要人类以批判性视角验证 AI 的输出或决策。

**策略：**
1. AI 完成某阶段工作产出结果
2. 将结果提交给人类，**明确要求人类挑错、找边界 case、质疑假设**
3. AI 逐一处理人类反馈的每个问题
4. 处理完毕后再次提交验证
5. 人类连续两次无新问题 → 通过

**示例：**
```
目标：验证安全架构方案

AI 产出: 安全架构设计文档
human_action → "请以攻击者视角审查此安全方案，找出所有潜在漏洞。
               不要只说好的方面，只找问题。"

人类返回: 3 个潜在问题
  - SQL 注入风险：参数验证不完整
  - Token 过期策略有漏洞
  - 日志中可能泄露敏感字段

AI 逐一修复 → 生成 v2 版本
human_action → "针对 v2 版本，再次以攻击者视角审查，只找问题。"

人类返回: 无新增问题

AI 标记验证通过
```

**使用要点：**
- 明确要求人类扮演"挑错者"角色（而非"审查者"或"确认者"）
- 指令中强调"只找问题，不要说好的方面"
- 每次人类反馈都必须逐一回应
- 连续两轮无新问题方可通过
- 适用于安全审查、架构评审、关键业务逻辑验证

## AI 验证模式

当人类返回结果后，AI **必须**对结果进行验证，不得盲目信任。验证从以下四个维度进行：

### 1. 格式验证（Format Check）
**检查内容：**
- 返回的数据类型是否符合预期（字符串/数字/布尔/列表/JSON）
- 必填字段是否完整
- 数值是否在合理范围内
- 文件路径格式是否正确
- 日期时间格式是否符合 ISO8601

**不通过时的处理：**
- 如果格式问题不影响语义理解，AI 自行解析/转换
- 如果格式严重偏离导致无法解析，通过 `human_information` 请求人类按指定格式重新提供（新 task）

### 2. 一致性验证（Consistency Check）
**检查内容：**
- 同一次返回中是否存在自相矛盾的信息
- 与已知事实/文档/代码库是否一致
- 与历史任务结果是否矛盾
- 多个并行任务返回的结果之间是否一致（Fan-Out 场景）

**不通过时的处理：**
- 标记矛盾点
- AI 根据可信度权重判断哪一方更可靠
- 如果无法判断，通过 `human_decision` 请人类澄清矛盾

### 3. 完整性验证（Completeness Check）
**检查内容：**
- 是否回答了原问题的所有子问题
- 是否有"TODO"、"待补充"、"不清楚"等占位符
- 边界情况是否被考虑
- 异常情况是否被说明

**不通过时的处理：**
- 列出遗漏项
- 通过 `human_information` 一次性追问所有遗漏项（新 task，附上原始问题上下文）

### 4. 合理性验证（Reasonableness Check）
**检查内容：**
- 结果是否符合技术常识和逻辑
- 数值是否在合理量级（如"数据库有 100TB"就明显不合理）
- 时间估算是否现实
- 依赖关系是否成立
- 是否存在明显的误解（人类可能误解了 AI 的问题）

**不通过时的处理：**
- 指出具体的不合理之处
- 通过 `human_information` 或 `human_decision` 与人类确认（带上 AI 的推理过程）
- 如果确认人类结果有误，附上正确信息请求修正

### 验证决策树

```
收到人类结果
    │
    ├── 格式验证 ── 不通过 ──▶ AI 尝试修复/解析
    │                              │
    │                              ├── 可修复 ──▶ 使用修复后的结果
    │                              └── 不可修复 ──▶ 请求重新提供
    │
    ├── 一致性验证 ── 不通过 ──▶ 标记矛盾 → 请求澄清
    │
    ├── 完整性验证 ── 不通过 ──▶ 列出遗漏 → 追问补充
    │
    ├── 合理性验证 ── 不通过 ──▶ 指出疑点 → 确认修正
    │
    └── 全部通过 ──▶ 采纳结果，继续执行
```

## 优先级指南

### `critical` — 紧急
**使用场景：**
- 线上事故处理（如服务宕机、数据丢失）
- 安全事件响应（如数据泄露、攻击进行中）
- 阻塞全部后续工作的关键路径任务
- 有硬性外部 deadline 且已临近的任务

**人类端期望响应时间：** 分钟级
**AI 行为：**
- 集中所有注意力推动该任务
- 可中断其他非 critical 任务
- 到达 deadline 后立即升级

### `high` — 重要
**使用场景：**
- 当前工作流的重要依赖（不完成则无法继续当前阶段）
- 有时效性但不紧急的任务（如今天内需要用户确认）
- 影响多人工作进度的阻塞项

**人类端期望响应时间：** 小时级
**AI 行为：**
- 优先处理但可在等待期间做其他事
- Fan-Out 中标记为高优先级，要求先于 normal 任务处理
- 超过 deadline 2 小时后仍未完成，考虑升级

### `normal` — 常规
**使用场景：**
- 大多数日常任务（如信息收集、常规确认）
- 有 deadline 但不紧迫
- 可并行处理且不阻塞主流程的任务

**人类端期望响应时间：** 当天
**AI 行为：**
- 默认优先级
- 用 Fan-Out 批量处理以提高效率
- 到 deadline 仍未完成时，评估是否可以继续等待

### `low` — 低优先级
**使用场景：**
- "有空再做"的补充性任务
- 锦上添花而非必需的操作
- 探索性的信息收集
- 可选优化项

**人类端期望响应时间：** 本周内
**AI 行为：**
- 不阻塞任何流程
- 可在多个低优先级任务超过一定数量后批量提醒
- 超时后自动取消（不升级）

### 优先级升级路径

```
low (超时) → 自动取消
normal (超时) → AI 评估 → 提升为 high 或取消
high (超时) → 提升为 critical 或切换替代方案
critical (超时) → 持续升级，寻找替代人类操作员
```

## 规则

### 规则 1：永远不要委托 AI 自己能做的事
在调用任何 human_* 工具之前，先问自己：
- 这个操作能通过本地命令行完成吗？
- 这个信息能通过搜索或读取文件获取吗？
- 这个决策能基于现有规则和数据分析做出吗？

如果答案是"能"，就不要浪费人类时间。HumanMCP 是 AI 的能力边界扩展，不是 AI 的偷懒工具。

### 规则 2：提供完整上下文，让人类无需搜索
人类操作员不像 AI 那样可以瞬间搜索代码库、查阅文档。每次委托时：
- 附上所有相关的文件路径、行号、代码片段
- 说明任务背景和目标（不只是"做什么"，还有"为什么"）
- 给出具体的操作步骤（不要假设人类知道怎么做）
- 给出期望的输出格式（不要让人猜你想要什么）

反例：「帮我看一下那个报错是怎么回事」
正例：「在执行 `p4 submit -c 12345` 时出现 `Connection refused` 错误。
       当前 P4PORT=perforce.dj01.com:1666。
       请确认：(1) P4 服务器是否正常运行？
       (2) 当前 IP 是否在防火墙白名单中？
       (3) 网络代理是否需要配置？」

### 规则 3：使用 priority 和 deadline 帮助人类排优先级
- 每个任务都必须指定 priority 和 deadline
- deadline 必须合理（不要设 1 分钟 deadline 做需要 1 小时的事）
- 多个任务时，给出相对优先级顺序
- 如果 deadline 需要调整，主动告知人类

### 规则 4：及时取消不再需要的任务
如果任务的依赖条件发生变化（如方案变更、问题被解决、不再需要某信息），立即使用 `human_cancel` 取消：
```
human_cancel(task_id="task_20260602_001", reason="方案已变更，不再需要该信息")
```
不要让人类浪费时间去完成已经无意义的任务。

### 规则 5：人类不审查 AI 的工作（不存在 human_review 工具）
HumanMCP 协议中**没有 `human_review` 工具**。意思是：
- AI 不应该把代码改动发给人类请求审查
- AI 不应该把文档/方案发给人类请求"看看有没有问题"
- AI 对自己的产出质量负责

如果需要人类验证特定问题，使用 Adversarial Verify 模式，明确指定：
- 审查的具体维度（安全性？性能？正确性？）
- 具体的审查目标（找漏洞？找边界 case？核验数据？）
- 而非笼统的"帮我 review 一下"

### 规则 6：人类是 AI 工具箱中的工具，不是 AI 的主管
- AI 是任务的主导者和决策者
- 人类执行 AI 无法执行的操作
- AI 不应该向人类"请示"——这混淆了角色
- 需要决策时，使用 `human_decision` 让人类在 AI 准备好的选项中选择，而非直接问人类"我该怎么办"
- AI 不应该将困难和不确定性转嫁给人类

### 规则 7：任务结束后闭环反馈
当人类完成的任务产生了有价值的结论或产出，AI 应：
- 总结结果并记录在适当的地方（如代码注释、文档、MEMORY.md）
- 如果结果推翻了之前的假设，更新相关记录
- 如果结果产生了新的可复用知识，沉淀到知识库

### 规则 8：异步不阻塞
- 发起 human_* 任务后立即异步处理，不阻塞等待
- 利用等待时间处理其他独立任务
- 除非任务间有严格依赖（Pipeline 模式），否则不要让人类成为瓶颈
- 如果一个工作流中有 3 个人类任务和 5 个 AI 任务，优先启动所有人类任务，然后在等待期间完成 AI 任务

## 快速参考卡片

| 工具 | 一句话用途 | 关键触发词 |
|------|----------|-----------|
| `human_action` | 人类执行操作/命令/协调 | 物理操作、外部系统 GUI、受限命令、跨团队协作 |
| `human_decision` | 人类做主观选择 | 审美判断、业务决策、伦理权衡 |
| `human_information` | 向人类要信息/查系统 | 隐性知识、离线数据、封闭系统查询 |
| `human_poll` | 查询任务结果（单个/批量） | 轮询、检查状态 |
| `human_cancel` | 取消/修改未完成任务 | 取消任务、修改任务 |
| `human_list_tasks` | 查看队列 | 队列、待处理列表 |

| 状态 | 含义 | AI 应做 |
|------|------|--------|
| `pending` | 等待人类处理 | 继续其他工作，定时轮询 |
| `completed` | 任务完成 | **验证结果**（格式/一致性/完整性/合理性） |
| `rejected` | 任务被拒 | 读取原因 → 换路径，绝不重试 |

| 拒绝原因 | 含义 | AI 必须 |
|---------|------|---------|
| `ai_can_do` | AI自己能做 | 自己完成，不重试 |
| `unclear` | 指令不清 | 补充上下文后新建 task，不重试 |
| `out_of_scope` | 超出职责/权限 | 协调或升级，不重试 |
| `invalid_task` | 前提有误 | 修正前提，不重试 |
