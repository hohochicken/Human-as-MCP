# Human-as-MCP — 设计文档

## 1. 核心设计哲学

```
┌──────────────────────────────────────────────────────────────┐
│  AI 是编排者 + 验证者，人是 AI 工具箱中的一个"工具"            │
│                                                              │
│  ✅ AI 做它能做的所有事 → 遇到能力/权限边界 → 调用人类工具      │
│  ✅ AI 审核人类返回的结果 → 验证合理性 → 整合到工作流           │
│  ✅ 人类只负责提供 AI 确实没有的能力，不需要理解完整上下文        │
│  ❌ 人类不要审核 AI 的日常产出（重大方向决定除外）               │
└──────────────────────────────────────────────────────────────┘
```

**关键区别：**
- 错误思维：AI 干活 → 人检查 → AI 修改 → 人再检查（人是 AI 的 supervisor）
- 正确思维：AI 编排全天工作 → 遇到自己做不了的 → 调用人 → 人执行并返回 → **AI 审核人的返回** → AI 继续编排

## 3. 工具定义（6 个工具）

### 工具全景

| 工具 | 用途 | AI 做不到的原因 |
|------|------|---------------|
| `human_action` | 执行手动操作、运行受限命令、人际协调 | AI 没有操作真实软件/设备的能力，无法主动联系人 |
| `human_decision` | 拍板重大方向（架构、资源、风险） | 涉及商业判断和风险承担 |
| `human_information` | 提供未文档化的知识、查询封闭系统 | 信息只存在人脑中，或系统没有对 AI 开放的 API |
| `human_poll` | 查询任务结果（单个或批量） | — |
| `human_cancel` | AI 取消/修改未完成的任务 | — |
| `human_list_tasks` | 查询当前队列情况 | — |

### 3.1 `human_action` — 手动操作

AI 让你做它碰不到的事。可附带明确步骤，也可只描述目标。

```
参数:
  title              string   简短标题
  description        string   完整描述（要做什么、期望什么结果）
  steps              string[] 可选。逐步操作说明，AI 能写多细写多细
  action_type        enum     operation | command | coordination | build | config | other
  priority           enum     low | normal | high | critical
  deadline_minutes   int      超时后任务保持 pending，AI 可重新提醒

返回（人类在 Dashboard 填写）:
  result             string   执行结果描述
  evidence           string[] 可选。证据附件（截图路径、日志片段等）
```

### 3.3 `human_information` — 提供未文档化知识与系统查询

AI 需要的信息不在代码库/文档中，或存在于封闭系统内。

```
参数:
  question           string   精确的问题 或 查询内容
  context            string   AI 为什么需要、会怎么用
  source             enum     person | system    知识来源：人的经验 或 封闭系统查询
  domain             enum     architecture | design | history | convention | workflow | contact | other
  priority           enum
  deadline_minutes   int

返回:
  answer             string   回答 或 查询数据
  confidence         enum     high | medium | low
  sources            string[] 可选。信息来源 或 来源链接
```

### 3.6 `human_decision` — 重大方向拍板

```
参数:
  title              string   决定标题
  context            string   AI 已做的完整分析和背景
  options            string[] 可选项
  recommendation     string   AI 推荐哪个选项 + 理由
  decision_type      enum     architecture | resource | risk | direction | other
  impact             string   决定的影响范围
  priority           enum
  deadline_minutes   int

返回:
  chosen             string   选择的选项
  reasoning          string   可选。选择理由
```

**触发阈值：** 提交代码、选择架构方案、需要大规模修改工程内容。其他日常操作 AI 自己决定。

### 基础设施工具

**`human_poll`** — 查询任务状态（支持单个 ID 或 ID 列表）
```
参数: task_id  或  task_ids[]
返回单个: { status: pending|completed|rejected, result?: ..., rejected_reason?: ... }
返回批量: [{ task_id, status, result?, rejected_reason? }, ...]
```

**`human_cancel`** — AI 取消/修改未完成的任务
```
参数: task_id, action: cancel | modify, new_data?: {...}
行为: Server 标记为"待确认"，人在 Dashboard 确认后生效。修改支持改描述、改优先级
```

**`human_list_tasks`** — 查询当前队列
```
参数: status (可选，筛选状态), limit (可选，最大返回数)
返回: { tasks: [{ task_id, title, status, priority, created_at, deadline, ... }], total_count }
```

## 4. 任务生命周期

### 状态机

```
                    AI 创建任务
                         │
                         ▼
                    ┌─────────┐
                    │ pending  │  ← 任务自动指派给人，无需 accept
                    └────┬────┘
                         │
              ┌──────────┼──────────┐
              ▼                     ▼
         ┌──────────┐         ┌──────────┐
         │ completed │         │ rejected  │
         └──────────┘         └─────┬────┘
               │                     │
               │              拒绝理由（4 种）:
               │              • ai_can_do     — AI 自己能做
               │              • unclear       — 描述不清楚
               │              • out_of_scope  — 超出能力/职责
               │              • invalid_task  — 前提有误
               │                     │
               ▼                     ▼
         任务结束，AI 审核      全部含义：AI 你自己想办法，不要重试
```

**没有 `accepted`/`in_progress` 状态。** 任务到达即为人的责任，只有完成和拒绝两个出口。

### 同步阻塞 + 异步回退（3 分钟规则）

AI 调用人类工具后，Server 会**阻塞等待最多 180 秒**，每 3 秒轮询一次任务状态：

```
AI 调用 human_action(...)
  → Server 创建任务 + Toast 通知
  → 每 3 秒检查任务是否 completed/rejected
  → 180 秒内人类在 Dashboard 完成/拒绝
      → ✅ 同步返回结果给 AI（sync: true）
  → 180 秒超时人类未响应
      → ⏳ 回退异步模式，返回 task_id（sync: false）
      → AI 后续通过 human_poll 轮询
```

**设计意图：** 如果人类正好在电脑前、看到 Toast 后立即处理，AI 可以同步拿到结果继续工作流，不需要额外的轮询调用。180 秒后仍未响应说明人类不在或正在忙，任务自然成为异步看板任务。

### 超时处理

`deadline_minutes` 到期 → 任务保持 `pending`，不自动取消。AI 可以主动 re-ping（通过 `human_cancel` 修改任务并重新提醒），或继续异步等待。

### 并发模型

AI 可以同时派发多个任务，不限制并发数。人按照任务优先级自行决定执行顺序。Dashboard 按优先级倒序排列。

## 5. AI 审核人类返回的模式

每个 `human_poll` 拿到结果后，AI 执行四步验证：

```
1. 格式检查    — 返回是否符合预期 schema？必填字段是否都有？
2. 一致性检查  — 返回的信息和 AI 已知信息有矛盾吗？
3. 完整性检查  — 期望的信息都提供了吗？
4. 合理性检查  — 人类说的事情从技术角度合理吗？
                  → 有问题 → 起新 human_information 追问澄清
                  → 没问题 → 整合结果，继续工作流
```

这是 AI 作为编排者正常验证任何工具返回的行为——人对 AI 来说就是一个返回结果可能不准的工具，需要验证。

## 7. Workflow 模式

### Fan-Out（最常用）
AI 同时派发多个独立人类任务 → 收集 task_id → `human_poll`（批量）轮询 → 谁先完成先用谁的结果 → 审核验证 → 继续编排

### Pipeline（人类是其中一环）
AI 准备 → `human_action` → 审核人类返回 → AI 继续处理 → 产出最终结果

### Loop-Until-Clear（澄清循环）
AI 调用 `human_information` → 审核返回 → 发现不一致/不完整 → 再起 `human_information` 追问 → 直到信息自洽

### Adversarial Verify（人类做最终验证）
AI 分析完成 → `human_decision`（"这个分析正确吗？方案可行吗？"）→ 人拍板 → AI 按照决定继续实现

## 8. 多 Agent 场景

多个 AI Agent（Claude Code、HermesAgent 等）共享同一个 Human-as-MCP Server。

**队列策略：** 所有 Agent 的任务进入同一个队列，按优先级统一排序，不区分来源。人在 Dashboard 看到的是一个合并后的列表，不需要知道任务来自哪个 Agent。

**AI 取消/修改时确认：** 只有发起任务的 Agent 能取消/修改该任务。取消/修改操作进入"待确认"状态，人确认后生效。

---

## 9. 边界与安全

**信任 AI，仅限制频率。** 不对任务内容做系统级审查。仅限制：
- 单个 Agent 每小时最大任务数
- 全局每小时最大任务数

具体数字通过配置文件调整，初始默认值：每 Agent 30 任务/小时，全局 100 任务/小时。

## 10. 设计决策汇总

| # | 决策 | 选择 |
|---|------|------|
| 1 | 人类角色 | AI 的工具，不是 AI 的 supervisor |
| 2 | 审核方向 | AI 审核人的返回结果 |
| 3 | 交互模式 | 单次往返，一次任务 = 一轮交互。追问 = 新任务 |
| 4 | 任务步骤 | `steps` 可选，AI 能写多细写多细 |
| 5 | 返回证据 | `evidence` 可选，人自行判断 |
| 6 | AI 等待策略 | 所有工具统一阻塞 180 秒轮询，超时回退异步 |
| 7 | 任务承接 | 无需 accept，任务到达即自动负责 |
| 8 | 任务状态 | 仅 pending / completed / rejected，无中间态 |
| 9 | 拒绝理由 | 4 种：ai_can_do / unclear / out_of_scope / invalid_task |
| 10 | 拒绝后 AI 行为 | 全部意味着 AI 别重试，自己想办法 |
| 11 | 超时处理 | 保持 pending，AI 可主动 re-ping |
| 12 | 优先级管理 | AI 全权排序，人看 Dashboard 顶部 |
| 13 | 并发派发 | AI 可同时派多个，人自定执行顺序 |
| 14 | AI 取消/修改任务 | AI 可发起，人确认后生效 |
| 15 | 多 Agent | 合并透明队列，不区分来源 |
| 16 | 通知 | Windows Toast，点击打开浏览器 Dashboard |
| 17 | Dashboard | 内嵌于 MCP Server，localhost:4350/dashboard |
| 18 | Dashboard 结构 | 两个 Tab：待处理 + 历史（按日期分组） |
| 19 | Dashboard 刷新 | WebSocket 实时推送 |
| 20 | 历史信息密度 | 标题 + 完成时间 + 结果/拒绝理由摘要 |
| 21 | 决策阈值 | 提交代码、选架构方案、大规模修改 |
| 22 | 审批信息量 | 一句话摘要（变更号+标题+文件数+行数变化） |
| 23 | 协调闭环 | 单次往返，部分结果也算完成 |
| 24 | 离线处理 | 任务照常排队，第二天消化 |
| 25 | 安全策略 | 仅频率限制，信任 AI |
| 26 | human_review | 删除，人不管 AI 的产出 |
| 27 | human_approve | 合并到 human_decision |
| 28 | 同步阻塞 | AI 调用后阻塞等待最多 180 秒，人类即时响应则同步返回；超时回退异步 task_id 模式 |
